# Loop+ Agnostic Guide
_Version: 2026-03-29bb_

## Purpose
Loop+ is the outer research-and-integration loop for any serious project.
It is intentionally project-agnostic.

It exists to:
- widen search intelligently
- prevent narrow local maxima
- turn outside signal into usable internal pressure
- keep research tied to build and verification
- reduce drift, rediscovery, and folklore

Loop+ is not a replacement for implementation, experimentation, or control docs.
It is the disciplined outer loop that feeds them.

---

## 1. What Loop+ Is

Loop+ is:

1. **Primary-source research**
2. **Discovery expansion**
3. **Fast-signal / implementation-clue scanning**
4. **Isomorph hunting**
5. **Native translation**
6. **Control-layer update**
7. **Experiment pressure**
8. **Recursive replay**

It is designed to sit outside the core internal loop:

> **PROBE → DERIVE → EMBODY → VERIFY → RECURSE**

The inner loop turns evidence into build pressure.  
Loop+ makes sure the evidence frontier stays alive, broad, and relevant.

---

## 2. Core Principles

### 2.1 Research must serve the build
Research that does not affect:
- architecture
- benchmarks
- queue ordering
- risk model
- control docs
- or active seams

is background, not Loop+.

### 2.2 Outside jargon must be translated
Do not leave findings in source-native terminology if they are going to affect the project.
Translate them into:
- the project’s ontology
- variable names
- queue effects
- concrete experiments
- doctrine changes

### 2.3 One paper is not a doctrine
Representative evidence matters more than isolated excitement.
Use groups of findings to sharpen a seam, not single papers to declare truth.

### 2.4 Research is not allowed to float free
Every Loop+ pass should end in one of:
- a new experiment
- a revised baseline
- a refined seam
- a demoted assumption
- a promoted doctrine rule
- or a decision not to spend compute there

### 2.5 Recursion is mandatory
If a new finding changes interpretation materially, replay earlier results under the new frame.

## 3. The Loop+ Stages

## Stage 1 — Primary-source pass
Start with the highest-signal sources available:
- papers
- technical reports
- standards
- official docs
- benchmark methodology
- discipline-specific repositories

Goal:
- find mechanisms
- find constraints
- find formal definitions
- find what the field actually claims, not what social media says it claims

Output:
- source list
- short notes by seam
- candidate mechanisms / constraints

---

## Stage 2 — Discovery expansion
Use discovery tools to widen from the seed set:
- citation chasing
- repository search
- related-work graphs
- adjacent-domain search
- keyword evolution

Goal:
- surface nearby work with different vocabulary
- prevent term-lock
- find the seam’s hidden neighborhood

Output:
- expanded source graph
- alternative terminology
- adjacent domains
- possible isomorph candidates

---

## Stage 3 — Fast-signal / implementation-clue pass
Scan practical / faster-moving sources:
- engineering news
- implementation blogs
- release notes
- hardware / infra reviews
- benchmark-heavy outlets
- practitioner discussion when necessary

Goal:
- catch deployment clues
- catch implementation constraints
- catch infrastructure shifts
- catch “this may suddenly matter now” signals

Rule:
These are not primary truth sources for theory.
They are pressure applicators and implementation scent trails.

Output:
- practical clues
- infra/hardware relevance
- candidate “pressure” items
- timing / cost / feasibility notes

---

## Stage 4 — Isomorph-predator pass
Take the seam and hunt structural equivalents elsewhere.

Ask:
- what is the same shape in another field?
- what is the equivalent in signal processing?
- what is the equivalent in retrieval/memory?
- what is the equivalent in control systems?
- what is the equivalent in geometry, compression, routing, or QA?

Goal:
- escape vocabulary traps
- import mature patterns without importing baggage blindly
- discover higher-order structure

Output:
- isomorphism list
- candidate transfers
- caution flags for bad analogies

---

## Stage 5 — Native translation
Translate the research back into project-neutral or project-native operational terms.

For each promising finding, state:
- what seam it touches
- what variable it changes
- what assumption it strengthens or weakens
- what benchmark or control doc it should affect
- whether it is:
  - new signal
  - reinforcement
  - contradiction
  - correction
  - or trivia

Output:
- translated seam notes
- variable map
- doctrine deltas
- candidate experiment queue entries

---

## Stage 6 — Control-layer update
Research must enter the project memory.

Recommended updates:
- research crosswalk
- trace matrix
- doctrine snapshot
- next steps
- revisit ledger
- ingest / extraction package when the source set is large

Goal:
- prevent orphan insight
- preserve provenance
- make restart/recovery possible
- keep current state honest

Output:
- updated control docs
- delta note
- promoted vs provisional findings

---

## Stage 7 — Experiment pressure
Convert the best findings into direct pressure on the build.

Examples:
- new baseline comparison
- variable isolation test
- ablation
- inversion / wrong-looking branch
- structural distinctness check
- teacher/student compression test
- retrieval or memory bridge test

Goal:
- turn outside knowledge into local evidence

Output:
- experiment spec
- ordering in queue
- preregistration if confirmatory
- expected kill criterion

---

## Stage 8 — Recursive replay
After a meaningful result:
- replay earlier seams under the new interpretation
- re-evaluate prior “dead” paths
- reclassify active/incumbent/collapsed
- update the control layer again

Goal:
- stop the project from accumulating stale negatives
- stop old interpretations from silently surviving after the assumptions beneath them changed

Output:
- replay report
- revised doctrine state
- revised queue

---

## 4. Source Buckets

Loop+ should treat sources as buckets with different epistemic roles.

### Bucket A — Primary truth sources
- papers
- standards
- official technical docs
- benchmark methodology
- formal reports

Use for:
- mechanisms
- definitions
- validated claims
- constraints

### Bucket B — Discovery infrastructure
- scholarly search engines
- repositories
- citation graphs
- archive aggregators

Use for:
- widening the search frontier
- finding synonyms
- finding adjacent work

### Bucket C — Fast-signal sources
- engineering/news outlets
- reviews
- release notes
- practitioner summaries

Use for:
- timing
- implementation clues
- infrastructure pressure
- “this may matter now” alerts

### Bucket D — Internal corpus
- uploaded documents
- extraction packages
- logs
- prior experiments
- control docs
- archived reports

Use for:
- local continuity
- avoiding duplication
- checking whether a “new” finding is actually old news

---

## 5. Output Discipline

A real Loop+ pass should leave behind artifacts.

Recommended artifact set:
- Loop+ report
- source manifest
- translated seam notes
- promoted/provisional findings list
- next-step pressure list
- optional extraction package for large corpora

If the pass matters, it should leave a file.

---

## 6. Failure Modes Loop+ Is Meant to Prevent

- paper hoarding without integration
- rediscovering the same seam under a new label
- term lock
- hype-driven architecture pivots
- letting fast-signal sources masquerade as theory
- orphaned outside insights
- stale negatives that survived assumption changes
- local maxima caused by too-narrow search

---

## 7. Minimal Loop+ Checklist

- [ ] Did we start from primary sources?
- [ ] Did we widen the search frontier?
- [ ] Did we scan fast practical signal?
- [ ] Did we hunt isomorphs?
- [ ] Did we translate the findings into operational terms?
- [ ] Did we update the control layer?
- [ ] Did we pressure the experiment queue?
- [ ] Did we replay earlier seams if assumptions changed?

---

## 8. Short Form

Loop+ is:

> **Research broadly, translate locally, pressure experiments, update memory, replay recursively.**

That is the agnostic version.
