# Universal Project SOP
_Version: 2026-03-29ax_

## Purpose
This SOP is a project-agnostic lab/process starter kit.
It is designed to be dropped into a new project thread so the work begins with:
- disciplined recordkeeping
- explicit rehydration rules
- zero-loss upload handling
- controlled doctrine updates
- recursive validation
- anti-drift safeguards

It is not tied to any specific architecture, domain, or codebase.

---

## 1. Load-First Rule
Before substantive project work:
1. Read the rehydration note.
2. Read this SOP.
3. Read the current state / next steps file if one exists.
4. Read the latest corpus/extraction docs if uploads exist.
5. Re-state:
   - incumbent baseline
   - open seams
   - what is actually verified
   - what is only provisional

Never trust recent chat memory alone.

---

## 2. Core Loop
**PROBE → DERIVE → EMBODY → VERIFY → RECURSE**

### Probe
Interrogate the problem, corpus, or system until structure, dependencies, contradictions, and failure modes are visible.

### Derive
Translate findings into falsifiable interpretations:
- what changed
- what did not
- what strengthened
- what weakened
- what now matters more

### Embody
Instantiate the interpretation into:
- docs
- architecture
- experiments
- process updates
- retrieval/index state
- next-step queue

### Verify
Do not promote without evidence.
Verification should include:
- direct test
- baseline comparison
- structural distinctness checks
- artifact generation
- explicit residuals / uncertainty

### Recurse
Replay earlier seams under new findings.
Do not assume earlier negatives remain informative after later discoveries.

---

## 3. Status Vocabulary
- **Open**: not explored enough
- **Active**: under current investigation
- **Incumbent**: current best-known baseline, still provisional
- **Collapsed**: sufficiently exhausted; justified closure

### Closure rule
A path is only collapsed when:
1. it was tested directly
2. nearby combinations were tested
3. inverted / wrong-looking versions were tested
4. recursive reruns happened after later findings
5. it still failed to improve or only gave redundancy

One failed framing does not close a variable class.

---

## 4. Zero-Loss Upload SOP
Treat meaningful uploads as lab evidence by default.

### Phase A — ELT ingest
- preserve raw contents
- create manifest
- hash artifacts
- map corpus shape
- do not destroy source structure before interpretation

### Phase B — OIE extraction
Normalize:
- claims
- relations
- dependencies
- process patterns
- contradictions
- implicit assumptions

### Phase C — Standardized extraction matrix
For each meaningful claim:
- source
- claim
- evidence strength
- doctrine effect
- status
- next action

### Phase D — Post-processing
Rescue:
- orphan signal
- hidden dependencies
- baseline mistakes
- category errors
- duplicated “open” questions
- contradictions

### Phase E — Control-layer integration
Promote only what is:
- repeated
- load-bearing
- architecture/process-shaping
- implementation-relevant
- or governance-critical

### Required output package
When justified, emit:
- manifest
- extraction doc
- triples/facts table
- standardized matrix
- synthesis
- package zip

---

## 5. Control Layer
Maintain a persistent control layer for any nontrivial project.

Recommended files:
- current doc set / index
- next steps
- SOP cycle update
- doctrine snapshot
- revisit ledger / supplement
- trace matrix
- research crosswalk
- maintenance note

### Rule
Update by delta, not amnesia.

---

## 6. Research Loop+
Use external research to pressure the build, not to decorate the chat.

Loop+:
1. primary-source pass
2. discovery pass
3. fast-signal / implementation-clue pass
4. isomorph-predator pass
5. native ontology translation
6. control-layer update
7. experiment pressure
8. recursive replay

Do not leave findings in outside jargon if they affect the project.

---

## 7. Anti-Drift Rules
- No silent lock-in
- No fake progress
- No orphan insight
- No benchmark theater
- No untracked doctrine changes
- No conflating distinct workstreams
- No treating runtime wins as automatic substrate wins
- No treating one summary as sufficient truth

---

## 8. Session Restart Procedure
If context is lost or compaction occurs:
1. stop widening
2. read rehydration note
3. read SOP
4. read current state / next steps
5. read doctrine snapshot
6. read relevant extraction docs
7. read active seam artifacts
8. re-state baseline vs open seams
9. continue

---

## 9. Minimum Artifact Discipline
Every meaningful experiment or extraction should leave artifacts behind.

Recommended outputs:
- spec
- report
- raw or summary table
- updated next steps
- updated doctrine/control delta

If it matters, it should leave a file.

---

## 10. Final Principle
This SOP exists so that “upload archive > continue” is enough to resume serious work without depending on fragile conversational memory.
