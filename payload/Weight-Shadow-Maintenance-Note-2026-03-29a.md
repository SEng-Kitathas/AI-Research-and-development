# Weight Substrate — Shadow Maintenance Note
_Version: 2026-03-29a_

## SINGLE RULE
If a branch changes, the matrix changes.
If a research seam changes, the crosswalk changes.
If the basis changes, the white paper changes.
If a genealogy source materially changes interpretation, the genealogy doc changes.
If none of those changed, the work probably did not matter.

## SESSION START ORDER
At the start of every meaningful session, load in this order:
1. Basis Document
2. Trace Matrix
3. Research Crosswalk
4. Addendum
5. Maintenance Note
6. White Paper Thesis
7. Source-Thread Genealogy

## NEW DOCUMENT DISCIPLINE
### When to update the genealogy doc
Update it when:
- a high-signal thread is read turn-by-turn
- an old thread’s role changes from speculative to load-bearing
- a thread is found to have created a branch or major law
- hype and grounded claims need to be separated

### When to update RB-5 rows
Update the matrix/crosswalk whenever:
- a new ingestion component is found
- scoring logic changes
- locality strategy changes
- export format changes
- provenance/CIL behavior changes

## BRANCH DISCIPLINE
Every branch should eventually have:
- claim
- object model
- operator family
- packing/encoding
- update rule
- task set
- metrics
- kill criteria
- ingestion/export assumptions (if applicable)

## PROHIBITIONS
Do not:
- collapse genealogy into doctrine
- treat Forge hype as equivalent to Forge code
- call RB-5 complete until native export exists
- let speculative scheduler/reversibility language creep back into the basis unmarked

## PRESERVATION RULES
- never delete shadow docs; supersede them
- preserve old branch names if part of genealogy
- preserve MHT threads as evidence inputs
- preserve zip/code artifacts as separate evidence from conversational claims
