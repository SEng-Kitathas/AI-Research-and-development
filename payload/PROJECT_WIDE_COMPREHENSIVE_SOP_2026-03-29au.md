# Project-Wide Comprehensive SOP
_Version: 2026-03-29au_

## 0. Purpose
This SOP exists to stop the project from dissolving into:
- chat drift
- compaction loss
- duplicated ideas
- forgotten requirements
- disconnected research
- silent assumption lock-in
- fake progress
- premature closure
- evidence debris
- untracked doctrine changes

This is not a note.  
It is the **project-wide operating system for the lab process**.

It governs:
- research
- corpus extraction
- shadow-doc maintenance
- experimentation
- doctrine promotion
- closure / non-closure
- recursive replay
- LLM rehydration after context loss

---

## 1. LLM Rehydration Preamble (Load This First)

### 1.1 First instruction to any LLM resuming this project
Before doing anything substantive, treat this project as a **lab**, not a casual conversation.

You are resuming a long-horizon architecture research and build process with:
- persistent shadow docs
- artifact corpus
- experiment history
- doctrine snapshots
- revisit ledgers
- extracted thread captures
- recursive process rules

Do **not** assume memory from chat context is sufficient.

### 1.2 Mandatory rehydration sequence
At the start of any meaningful project session, load in this order:

1. **This SOP**
2. **Current doc set**
3. **Latest next-steps doc**
4. **Latest SOP cycle update**
5. **Latest revisit ledger + supplement**
6. **Ontology / doctrine snapshot**
7. **Relevant extraction docs / uploaded corpus captures**
8. **Relevant experiment reports / CSVs for the active seam**

### 1.3 LLM danger-minimization rules
The LLM must assume the following failure modes are always possible:
- context compaction
- partial memory loss
- silent loss of earlier decisions
- overcompression of nuanced findings
- false confidence from recent context only
- treating incumbent baselines as closed doctrine
- merging runtime/project-A/project-B claims incorrectly
- treating research summaries as stronger than actual experiment evidence

To reduce loss:
- reload the control layer first
- prefer artifacts over memory
- prefer extraction docs over vague recollection
- prefer matrix/crosswalk/addendum/maintenance over narrative recall
- surface uncertainty explicitly
- never silently overwrite prior doctrine
- update by diff

### 1.4 Hard rule
If there is any sign of confusion, ambiguity, or process drift:
- stop widening
- reload the control docs
- re-establish incumbent baseline vs open seams
- then continue

## 2. Core Vocabulary

### 2.1 Status terms
- **Open**: not explored enough to justify closure
- **Active**: currently under investigation
- **Incumbent**: best current result; baseline to beat; still provisional
- **Collapsed**: only after the variable neighborhood has been sufficiently exhausted

### 2.2 Discipline terms
- **Zero-loss extraction**: preserve, extract, normalize, post-process, synthesize
- **Loop+**: external science/buzzword/isomorph research loop
- **Shadow layer**: persistent project-memory/control system
- **Monotonic atomicity**: new findings recurse backward through earlier seams
- **Isomorph predator**: hunt structural equivalents across domains
- **QA/evidence spine**: proof, risk, gate, assurance chain that advances with feature/build spine

## 3. Non-Negotiable Governing Rules

### Rule 1 — Nothing locks in early
No architecture, variable, or doctrine becomes final because it is currently winning.

### Rule 2 — Best current is not final
Incumbent means:
- best current operational baseline
- still provisional
- must survive future replay

### Rule 3 — No silent overwrite
Update by delta, not by amnesia.

### Rule 4 — No orphan insight
If a finding matters, it must land in:
- shadow docs
- a spec
- a report
- the doctrine snapshot
- or the revisit ledger

### Rule 5 — No fake progress
A long discussion does not count as advancement unless:
- docs moved
- artifacts were produced
- or a seam was genuinely narrowed

### Rule 6 — No disconnected research
Research must change:
- build pressure
- doctrine
- ordering
- or active seams
Otherwise it stays background.

### Rule 7 — No feature-only planning
The QA/evidence spine must advance with the feature/build spine.

### Rule 8 — No hidden uncertainty
If something is:
- partial
- residual
- speculative
- not yet wired
- not yet validated
say so explicitly.

### Rule 9 — No closure by one failed framing
A failed implementation kills that framing, not necessarily the variable class.

### Rule 10 — No single-source sufficiency
Do not treat any one of the following as sufficient by itself:
- current thread
- recent memory
- one upload
- one report
- one benchmark
- one paper
- one model’s opinion

---

## 4. Closure Standard

A path is only considered **collapsed** when all of the following have happened:

1. Direct test
2. Nearby-combination test
3. Inversion / wrong-looking test
4. Recursive replay after later findings
5. Still no meaningful gain beyond redundancy

Neutral once is not dead.
Bad framing is not variable closure.
Silence is not evidence.

## 5. The Master Internal Loop

## PROBE → DERIVE → EMBODY → VERIFY → RECURSE

### 5.1 Probe
Interrogate the system/corpus until you expose:
- actual structure
- hidden dependencies
- contradictions
- failure modes
- uncertainty
- baseline mistakes

### 5.2 Derive
Translate the probe result into a falsifiable interpretation:
- what changed
- what did not
- what this now implies
- what assumption got stronger or weaker

### 5.3 Embody
Instantiate that implication into:
- architecture
- runtime
- readout
- memory policy
- scheduler
- benchmark
- document set
- ontology
- closure rules
- next steps

### 5.4 Verify
Check whether embodiment cashes out:
- rerun
- compare
- diff
- measure
- record residuals
- update docs
- update incumbent vs open seams

### 5.5 Recurse
Repeat forward and backward.
If a new result meaningfully changes interpretation, rerun earlier seams under the new view.

## 9. Zero-Loss Upload SOP (Default for All Uploads)

All uploads are treated as lab evidence by default.

## 9.1 Phase A — ELT ingest
- preserve raw files
- create manifest
- hash artifacts
- map corpus
- do not destroy raw structure before interpretation

## 9.2 Phase B — OIE extraction
Normalize:
- entities
- relations
- claims
- process patterns
- dependencies
- doctrine changes
- benchmark implications

## 9.3 Phase C — Standardized extraction matrix
For each meaningful claim:
- source artifact
- claim
- evidence strength
- doctrine effect
- status
- next action

## 9.4 Phase D — DEFNLP-style post-processing
Rescue:
- orphan signal
- contradictions
- hidden dependencies
- baseline errors
- ambiguous terminology
- facts that would otherwise vanish in summary

## 9.5 Phase E — Control-layer integration
Promote only what is:
- repeated
- load-bearing
- architecture-shaping
- research-validated
- implementation-relevant
- or process-critical

## 9.6 Output package
Each serious upload should yield:
- manifest
- extraction doc
- triples/facts table if useful
- standardized matrix
- post-process synthesis
- package zip

## 10. Research SOP (Loop+)

Loop+ is the external science / discovery / buzzword / isomorph research loop layered on top of the internal loop.

## 10.1 Loop+ stages
1. **Primary-source pass**
2. **Discovery pass**
3. **Buzzword / fast-signal pass**
4. **Isomorph-predator pass**
5. **Native ontology translation**
6. **Control-layer update**
7. **Experiment pressure**
8. **Recursive replay**

## 10.2 Research rules
- group findings by seam, not by paper
- prefer representative sources over giant citation dumps
- record what the seam changes in the build
- record what it reinforces
- record uncertainty honestly
- do not collect trivia
- do not import outside jargon without translation

## 10.3 Source buckets
Primary science:
- arXiv
- discipline-specific preprint servers
- formal papers / technical docs / standards

Discovery:
- Semantic Scholar
- CORE
- repo discovery / citation chaining

Buzzword / fast-signal:
- engineering news / benchmark-heavy reviews / infra/product signal

Internal:
- uploaded captures
- extraction docs
- shadow docs
- experiment reports
- doctrine snapshots

### 11.1 No skim rule

Do not only search for terms.
Search for:
- structures
- motifs
- isomorphisms
- repeated process patterns
- same-shape ideas across different labels

### 11.3 Cross-domain rule
Map equivalent structures across:
- signal processing
- graphics
- sound
- memory
- routing
- control systems
- runtime assurance
- retrieval
- compression

### 11.4 Ground-truth rule
If compaction or memory conflict appears:
- trust artifacts over recollection
- trust extraction over vibes
- trust docs over narrative

## 13. Benchmark / Baseline SOP

### 13.1 Never trust one baseline
Use the correct floor for the task family.

### 13.2 Separate benchmark conditions
If prototype/reference quality matters, separate:
- bound-prototype
- estimated-prototype
- no-prototype
conditions explicitly

### 13.3 Domain split rule
Do not assume:
- geometric task results
transfer cleanly to
- text / sequence tasks

If text behaves differently, branch doctrine.

## 14. Doctrine Promotion SOP

A finding is promotable when it is:
- repeated
- load-bearing
- architecture-shaping
- research-supported
- implementation-relevant
- process-critical

A finding should stay provisional if it is:
- decorative
- speculative with no build consequence
- redundant
- not yet differentiated from preference
- not yet recursively replayed

## 15. Project-A / Project-B Separation Rule

Do not silently merge:
- runtime/reasoning wins
with
- weight-substrate validation

Project A and Project B may reinforce each other, but evidence for one is not automatic evidence for the other.

---

## 16. Current Incumbent Rule

The current incumbent stack may be used as:
- the baseline to beat
- the default comparison target
- the shortest operational summary

It may **not** be treated as:
- sacred architecture
- final doctrine
- closed truth
- reason to stop exploring nearby combinations

---

## 17. Compaction / Context-Loss Recovery Procedure

If context compaction or confusion occurs:

1. Stop widening
2. Load this SOP
3. Load current doc set
4. Load latest next steps
5. Load latest SOP cycle update
6. Load latest revisit ledger / supplement
7. Load ontology / doctrine snapshot
8. Load the relevant extraction docs and active-seam reports
9. Re-state:
   - incumbent baseline
   - open seams
   - recently promoted findings
   - what is actually closed vs still open
10. Only then resume
